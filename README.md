Auto Pulse: FPGA Hardware-in-the-Loop CI/CD
A Hardware-in-the-Loop (HIL) CI/CD framework that extends a software-based PWM manager into physical hardware validation. It uses an ESP32 as a test controller to dynamically request, measure, and verify real-world PWM signals generated by a Cyclone IV FPGA, creating a fully automated, closed-loop testing pipeline.

This pipeline bridges the gap between cloud infrastructure and physical desktop hardware. When a developer pushes code to GitHub, a local Self-Hosted Runner operating within a Windows 11 / WSL2 environment wakes up. It automatically simulates the Verilog, synthesizes the hardware bitstream via Quartus, leverages a Windows USB passthrough to flash the physical FPGA, and commands an ESP32 to execute physical verification on the raw silicon.

🔧 Hardware Requirements (The Physical Lab)
To execute the Continuous Deployment and HIL phases, the physical workstation requires the following components wired together:

The Device Under Test (DUT): An Intel Cyclone IV FPGA Development Board (EP4CE6E22C8).

The Programmer: Altera USB-Blaster cable to physically flash the .sof bitstream from the host machine to the FPGA.

The HIL Bridge Microcontroller: An ESP32 development board. This acts as the automated "tester" that sends SPI commands and calculates the high-resolution duty cycle of the returning PWM voltage.

The Local Server: A Windows 11 PC hosting a local GitHub Runner inside a WSL2 Ubuntu instance. It must remain powered on and connected to both hardware devices via USB.

Wiring: Standard jumper wires with a shared common ground:

3 wires for SPI (MOSI, SCLK, CS_N).

1 wire for the PWM signal output returning to the ESP32 GPIO interrupt pin.

(Verified via logic analyzer at 24 MS/s for signal integrity).

💻 Software Prerequisites (The CI/CD Stack)
1. Cloud & Orchestration
A GitHub repository with Actions enabled.

A GitHub Self-Hosted Runner agent installed and running as a background service inside the local WSL2 environment.

2. Windows 11 Host Environment
usbipd-win installed to bridge and attach physical USB targets (the Altera USB-Blaster and ESP32 UART) directly into the WSL2 virtual kernel.

3. Local Server / WSL2 Ubuntu
Python 3.12+ (The runner automatically generates an isolated .venv_runner using requirements.txt).

Python Libraries: cocotb, pytest (for simulation), and pyserial (for ESP32 communication).

Icarus Verilog (iverilog) and make for rapid logic simulation.

Intel Quartus Prime Lite Edition (CLI) installed in WSL for hardware synthesis (quartus_map, quartus_fit, quartus_asm).

Quartus Programmer (jtagd / quartus_pgm) configured to recognize the passed-through USB-Blaster.

PlatformIO Core CLI for building and flashing microcontroller targets directly from Linux.
